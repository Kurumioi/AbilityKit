# Shooter 示例远程版闭环实施计划

## 目标

把 shooter 示例从“本地联机/同步演示”补齐为“可远程运行的完整闭环”，支持：

- 一键登录
- 选择玩法并开始游戏
- 游戏中断线后重连恢复
- 玩法可定义结束条件
- 可配置玩家离线多久后清理房间

## 当前现状

### 已具备

- `SessionGrain` 已支持 guest 登录、session 校验、续期与登出。
- `RoomGrain` 已支持创建、加入、准备、选择英雄、开战和关闭。
- `BattleLogicHostGrain` 已支持玩法运行时装配、输入提交、状态同步与重连式状态推送。
- `ShooterRoomGameplayAdapter` 与 `ShooterBattleRuntimeAdapter` 已存在。
- 网关侧已有 `GuestLoginHandler`、`CreateRoomHandler`、`JoinRoomHandler`、`RoomReadyHandler`、`RoomPickHeroHandler`、`StartRoomBattleHandler`、`SubscribeStateSyncHandler`、`RequestFullStateSyncHandler`。

### 主要缺口

- 登录后没有统一的“恢复当前房间/战斗上下文”流程。
- `RoomGrain` 只维护成员集合，没有在线状态、最后活跃时间、离线宽限期与自动清理策略。
- shooter 玩法结束条件尚未形成统一的房间状态回收规则。
- 客户端缺少登录后自动判断“创建/加入/重连/继续战斗”的流程编排。

## 实施分解

### 1. 服务端会话与恢复

- 在 session 层保留当前账号的房间/战斗索引，支持登录后直接恢复上下文。
- 增加一个只读查询入口，返回账号当前所在房间、是否已开战、battleId、worldId、房间状态等。
- 登录成功后，客户端先走“恢复查询”，再决定进入大厅、重连房间或重连战斗。

### 2. 房间生命周期增强

- 在 `RoomGrain` 内引入成员在线状态、最后活跃时间和离线开始时间。
- 区分 `Join` / `Reconnect` / `LateJoin` / `Spectator` 之类的状态语义，避免把重连和首次加入混在一起。
- 把“玩家离线多长时间清理房间”做成房间标签或 `RoomSummary.Tags` 可配置项。
- 由房间定时检查离线成员并触发自动清理，或在房间关闭后进入延迟回收。

### 3. 玩法结束条件

- 在 shooter 玩法中定义结束条件来源：
  - 玩法内自然结束
  - 房间配置驱动结束
  - 战斗运行时返回结束信号
- 战斗结束后，明确房间是“保持结果态可回看”还是“立即关闭并回收”。
- 与房间目录联动，保证结束后列表状态正确刷新。

### 4. 客户端一键登录与重连流程

- 登录后先保存 `SessionToken` 与 `AccountId`。
- 启动时自动查询账号当前房间/战斗上下文。
- 如果未在局内：进入玩法选择或创建/加入房间流程。
- 如果已在未开战房间：直接回到房间大厅并恢复准备态。
- 如果已在战斗中：直接订阅状态同步并请求全量快照恢复。

### 5. 验收闭环

- 新账号登录后可创建房间并开局。
- 游戏中断线后重新登录，可回到同一房间或战斗。
- 战斗结束后重新登录，不会再误进旧战斗。
- 超过离线阈值后，房间会被自动清理。
- 房间满员、房间关闭、重复重连等边界场景行为明确。

## 推荐落地顺序

1. 先补“登录后恢复查询”与房间/战斗上下文索引。
2. 再补 `RoomGrain` 在线状态和离线清理策略。
3. 然后补客户端自动重连与流程编排。
4. 最后完善 shooter 玩法结束条件与验收测试。

## 备注

当前协议层已有基础房间与战斗流程入口，不建议先扩散协议字段；优先基于现有 `CreateRoom` / `JoinRoom` / `SetReady` / `StartBattle` / `SubscribeStateSync` 链路补齐恢复语义，再视需要补只读查询接口。

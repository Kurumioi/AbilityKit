## 1. 技能系统需要什么

### 1.1 技能系统的本质

```
┌─────────────────────────────────────────────────────────────────────┐
│                      一个技能系统需要什么？                             │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│   技能系统 = 回答一个问题：「什么时候，对谁，做什么」                    │
│                                                                      │
│   ┌─────────────┐    ┌─────────────┐    ┌─────────────┐           │
│   │  什么时候   │ →  │    对谁     │ →  │   做什么   │           │
│   │  (时机)     │    │   (目标)    │    │  (效果)     │           │
│   └─────────────┘    └─────────────┘    └─────────────┘           │
│                                                                      │
│   • 「什么时候」→ 玩家点击 / 被动触发 / 时间到了 / 条件满足          │
│   • 「对谁」→ 自己 / 目标 / 范围内的敌人 / 友军                    │
│   • 「做什么」→ 扣血 / 加血 / 加Buff / 位移 / 播放特效             │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

### 1.2 技能类型分类

从玩家视角，技能可以分为：

```
┌─────────────────────────────────────────────────────────────────────┐
│                         技能类型分类                                 │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│   主动技能                                                          │
│   ├── 点击即发（瞬发）                                              │
│   ├── 需要吟唱（引导）                                              │
│   ├── 需要瞄准（目标选择）                                          │
│   └── 组合技能                                                     │
│                                                                      │
│   被动技能                                                          │
│   ├── 永久生效                                                     │
│   ├── 条件触发                                                     │
│   └── 叠加触发                                                     │
│                                                                      │
│   持续效果（Buff/Debuff）                                           │
│   ├── 即时生效                                                     │
│   ├── 周期生效（DOT/HOT）                                          │
│   └── 触发型                                                       │
│                                                                      │
│   投射物                                                            │
│   ├── 直线弹道                                                     │
│   ├── 曲线弹道                                                     │
│   ├── 追踪弹道                                                     │
│   └── 分裂弹道                                                     │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

## 2. 概念澄清：常见设计误区

在设计技能系统时，经常会遇到一些「看似正确」但实际上容易导致架构混乱的概念。让我逐一分析这些误区，并给出问题的本质定义。

### 2.1 误区一：「万物皆 Buff」

**常见说法**：
> 「所有持续效果都应该建模为 Buff」

**问题分析**：
这种说法把「时间维度」和「效果载体」混淆了。

```
┌─────────────────────────────────────────────────────────────────────┐
│                      「万物皆 Buff」误区分析                          │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│   Buff 是什么？                                                      │
│   ┌─────────────────────────────────────────────────────────────┐  │
│   │  Buff = 时间管理 + 效果容器                                  │  │
│   │                                                             │  │
│   │  • 它解决的是「效果持续多久」的问题                          │  │
│   │  • 它是一个「载体」，里面可以装各种效果                      │  │
│   └─────────────────────────────────────────────────────────────┘  │
│                                                                      │
│   问题：                                                            │
│   ┌─────────────────────────────────────────────────────────────┐  │
│   │  1. 护盾是 Buff 吗？                                        │  │
│   │     → 护盾 = 吸收伤害的效果，不一定有时间限制                │  │
│   │                                                             │  │
│   │  2. 攻击力翻倍是 Buff 吗？                                   │  │
│   │     → 如果是「普攻时30%概率触发」，不需要时间管理            │  │
│   │                                                             │  │
│   │  3. 「击杀后刷新所有技能冷却」是 Buff 吗？                   │  │
│   │     → 这是事件响应，不是时间维度的效果                        │  │
│   └─────────────────────────────────────────────────────────────┘  │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

**正确理解**：

| 概念 | 本质 | 解决的问题 |
|------|------|----------|
| **Buff** | 时间维度的效果容器 | 效果持续多久？叠加规则？ |
| **被动触发** | 事件驱动的效果 | 什么时候触发？触发条件？ |
| **属性修改** | 数值的即时计算 | 怎么计算最终值？ |

**框架对应**：

```
┌─────────────────────────────────────────────────────────────────────┐
│                      框架的概念分层                                  │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│   效果（Effect）────────────── 做什么（扣血、加血、加Buff）         │
│       │                                                               │
│       ├── IEffectComponent     效果组件（属性修改、触发事件）        │
│       │                                                               │
│       └── GameplayEffectSpec  效果规格（组件的组合 + 生命周期）     │
│                                   │                                   │
│                                   ├── 瞬时（Instant）               │
│                                   ├── 持续（Duration） ← 这才是 Buff │
│                                   └── 永久（Infinite）               │
│                                                                      │
│   触发（Trigger）────────────── 什么时候做                          │
│       │                                                               │
│       ├── 主动触发（Pipeline）  玩家操作、时间轴                     │
│       └── 被动触发（Triggering）  事件监听、条件判断                │
│                                                                      │
│   调度（Schedule）───────────── 周期性地做                          │
│       │                                                               │
│       └── IScheduleManager    DOT/HOT、子弹飞行、心跳检测          │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

### 2.2 误区二：「万物皆组件」

**常见说法**：
> 「用组件模式，一切皆组件，Buff 是组件、触发器是组件、技能也是组件」

**问题分析**：
组件模式是好的，但「万物皆组件」会导致边界模糊，不知道什么时候该用组件，什么时候该用其他模式。

```
┌─────────────────────────────────────────────────────────────────────┐
│                    「万物皆组件」误区分析                              │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│   组件模式的本质是什么？                                              │
│   ┌─────────────────────────────────────────────────────────────┐  │
│   │  组件 = 可组合的功能单元                                      │  │
│   │                                                             │  │
│   │  好的组件：                                                  │  │
│   │  • 有明确的输入和输出                                        │  │
│   │  • 可以独立存在和测试                                        │  │
│   │  • 可以和其他组件自由组合                                    │  │
│   │                                                             │  │
│   │  滥用组件：                                                  │  │
│   │  • 把「流程控制」建模为组件（技能是组件？）                   │  │
│   │  • 把「条件判断」建模为组件（触发器是组件？）                 │  │
│   │  • 把「时间调度」建模为组件（Buff 是组件？）                 │  │
│   └─────────────────────────────────────────────────────────────┘  │
│                                                                      │
│   问题举例：                                                        │
│   ┌─────────────────────────────────────────────────────────────┐  │
│   │  「护盾组件」应该包含什么？                                  │  │
│   │                                                             │  │
│   │  方案A：护盾组件 = 属性修改 + 生命周期 + 伤害吸收逻辑        │  │
│   │  → 组件太大，承担了太多职责                                 │  │
│   │                                                             │  │
│   │  方案B：护盾组件 = 属性修改（仅此而已）                      │  │
│   │  → 组件太小，生命周期谁来管？                               │  │
│   │                                                             │  │
│   │  正确做法：                                                  │  │
│   │  ShieldEffectComponent（效果组件）                         │  │
│   │      +                                                        │  │
│   │  GameplayEffectSpec with Duration（效果规格，包含生命周期）  │  │
│   │      +                                                        │  │
│   │  DamageAbsorbComponent（独立组件，处理伤害吸收逻辑）         │  │
│   └─────────────────────────────────────────────────────────────┘  │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

**正确理解**：

```
┌─────────────────────────────────────────────────────────────────────┐
│                      组件的边界划分                                  │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│   IEffectComponent ─────────────── 效果单元                         │
│   • 只管「做什么」                                                 │
│   • 输入：EffectExecutionContext                                    │
│   • 输出：修改属性、触发事件                                        │
│   • 例子：AttributeEffectComponent、TriggerEventComponent         │
│                                                                      │
│   Phase ────────────────────────── 流程单元                         │
│   • 只管「什么时候做」                                             │
│   • 状态：Enter → Running → Complete                              │
│   • 例子：DurationalPhase、TimelinePhase                         │
│                                                                      │
│   Condition ────────────────────── 条件单元                        │
│   • 只管「判断是否满足」                                           │
│   • 输入：上下文                                                   │
│   • 输出：bool                                                     │
│   • 例子：CompareCondition、HasTagCondition                       │
│                                                                      │
│   IAction ───────────────────────── 行为单元                       │
│   • 只管「执行动作」                                               │
│   • 输入：上下文                                                   │
│   • 输出：执行结果                                                  │
│   • 例子：DamageAction、ApplyEffectAction                        │
│                                                                      │
│   原则：一个组件只解决一个问题                                      │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

### 2.3 误区三：AOE 概念混用

**常见说法**：
> 「AOE 就是范围效果，用一个 AOE 组件就行了」

**问题分析**：
AOE 实际上涉及多个不同的概念，很容易混用。

```
┌─────────────────────────────────────────────────────────────────────┐
│                        AOE 概念混用分析                              │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│   「AOE」实际上包含三个独立的问题：                                   │
│                                                                      │
│   ┌─────────────────────────────────────────────────────────────┐  │
│   │                                                             │  │
│   │   问题1：范围形状（Shape）                                  │  │
│   │   ─────────────────────                                    │  │
│   │   • 圆形：以某点为圆心，半径 R 的区域                       │  │
│   │   • 扇形：以某点为圆心，半径 R、角度 θ 的区域              │  │
│   │   • 矩形：长 L、宽 W 的矩形区域                             │  │
│   │   • 环形：内半径 R1、外半径 R2 的圆环                       │  │
│   │                                                             │  │
│   │   问题2：范围来源（Origin）                                 │  │
│   │   ─────────────────────                                    │  │
│   │   • 以施法者为中心                                          │  │
│   │   • 以目标为中心                                            │  │
│   │   • 以某点为中心（落点）                                    │  │
│   │   • 以某实体为中心                                          │  │
│   │                                                             │  │
│   │   问题3：检测时机（Timing）                                 │  │
│   │   ─────────────────────                                    │  │
│   │   • 瞬时检测：施法瞬间检测范围内的目标                       │  │
│   │   • 持续检测：效果持续期间持续检测范围内目标                 │  │
│   │   • 脉冲检测：每隔一段时间检测一次                           │  │
│   │                                                             │  │
│   └─────────────────────────────────────────────────────────────┘  │
│                                                                      │
│   常见错误：                                                        │
│   ┌─────────────────────────────────────────────────────────────┐  │
│   │  「AOE 组件」 = 圆形 + 以施法者为中心 + 瞬时检测            │  │
│   │                                                             │  │
│   │  问题：这个「组件」能处理「扇形」吗？                        │  │
│   │  问题：这个「组件」能处理「以目标为中心」吗？                │  │
│   │  问题：这个「组件」能处理「持续检测」吗？                     │  │
│   │                                                             │  │
│   │  答案：不能，因为三个问题是正交的，应该分开建模              │  │
│   └─────────────────────────────────────────────────────────────┘  │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

**正确理解**：

```
┌─────────────────────────────────────────────────────────────────────┐
│                        AOE 的正确拆分                                │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│   ┌─────────────────────────────────────────────────────────────┐  │
│   │                    AOE = 查找 + 应用                        │  │
│   │                                                             │  │
│   │   ┌──────────────┐    ┌──────────────┐                   │  │
│   │   │   TargetFinder │ → │   EffectApplicator │              │  │
│   │   │   目标查找    │    │    效果应用    │                   │  │
│   │   └──────┬───────┘    └──────────────┘                   │  │
│   │          │                                                    │  │
│   │          ▼                                                    │  │
│   │   ┌──────────────┐                                          │  │
│   │   │    Shape     │  ← 范围形状（圆形/扇形/矩形）           │  │
│   │   │   范围形状   │                                          │  │
│   │   └──────────────┘                                          │  │
│   │                                                             │  │
│   └─────────────────────────────────────────────────────────────┘  │
│                                                                      │
│   TargetFinder 的职责：                                             │
│   • 根据 Shape 找到范围内的目标                                    │
│   • 根据 Filter 过滤目标（阵营、职业、状态）                       │
│   • 根据 Selector 选择最终目标（全选/最近/血量最低）                │
│                                                                      │
│   EffectApplicator 的职责：                                         │
│   • 对每个目标应用效果                                              │
│   • 处理效果叠加规则                                                │
│   • 触发命中事件                                                   │
│                                                                      │
│   框架对应：                                                        │
│   • com.abilitykit.combat.targeting ── TargetFinder               │
│   • GameplayEffectSpec ── EffectApplicator                        │
│   • Shape 相关类 ── 范围形状定义                                   │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

### 2.4 误区四：「万物皆行为树」

**常见说法**：
> 「所有游戏逻辑都应该用行为树，AI 是行为树、技能也是行为树」

**问题分析**：
行为树是一个通用的决策工具，但它不是银弹。把它用在所有地方会导致性能问题和架构混乱。

```
┌─────────────────────────────────────────────────────────────────────┐
│                    「万物皆行为树」误区分析                            │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│   行为树适合的场景：                                                │
│   ┌─────────────────────────────────────────────────────────────┐  │
│   │  • 复杂的 AI 决策（巡逻、追击、逃跑、攻击策略）               │  │
│   │  • 状态转换复杂的流程（需要可视化的状态机）                   │  │
│   │  • 需要频繁调整的逻辑（策划配置优先）                         │  │
│   └─────────────────────────────────────────────────────────────┘  │
│                                                                      │
│   行为树不适合的场景：                                              │
│   ┌─────────────────────────────────────────────────────────────┐  │
│   │  • 简单的条件触发（受到伤害 → 反击）                         │  │
│   │    → 用 Triggering 更简洁                                    │  │
│   │                                                             │  │
│   │  • 固定流程的技能（施法 → 效果）                             │  │
│   │    → 用 Pipeline 更清晰                                      │  │
│   │                                                             │  │
│   │  • 高频调用的逻辑（每帧检测距离）                            │  │
│   │    → 用 Schedule 更高效                                      │  │
│   │                                                             │  │
│   │  • 需要精确时间控制的流程（时间轴事件）                       │  │
│   │    → 用 Timeline 更精确                                      │  │
│   └─────────────────────────────────────────────────────────────┘  │
│                                                                      │
│   性能对比：                                                        │
│   ┌─────────────────────────────────────────────────────────────┐  │
│   │  场景              │ 行为树       │ 框架方案              │  │
│   │  ─────────────────────────────────────────────────────────  │  │
│   │  被动触发         │ 每帧遍历判断  │ Triggering（事件驱动） │  │
│   │  技能施放         │ 树遍历执行   │ Pipeline（阶段执行）   │  │
│   │  DOT/HOT          │ 每帧更新状态  │ Schedule（定时调度）  │  │
│   └─────────────────────────────────────────────────────────────┘  │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

**正确理解**：

```
┌─────────────────────────────────────────────────────────────────────┐
│                      工具选择的决策树                                │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│                      ┌─────────────┐                               │
│                      │ 需要决策？   │                               │
│                      └──────┬──────┘                               │
│                             │                                       │
│              ┌─────────────┴─────────────┐                          │
│              ▼                           ▼                          │
│        ┌───────────┐              ┌───────────┐                   │
│        │   是      │              │   否      │                   │
```

## 附录：行业方案对比

在游戏行业，不同的项目采用了不同的技能系统设计思路。了解这些方案的优缺点，有助于理解为什么我们的框架采用当前的设计。

### A.1 主流方案对比

| 方案 | 代表项目 | 核心思路 | 优点 | 缺点 |
|------|----------|----------|------|------|
| **Dota2** | Dota2 | Event + Action + Modifier，Json/Lua 配置 | 灵活，可复用 Key | Key 过多时学习成本高，流程不直观 |
| **守望先锋** | Overwatch | Statescript 可视化节点图 | 直观，可视化编辑 | 对策划编程能力要求高 |
| **UE4 GAS** | 虚幻引擎 | Task + Effect + Blueprint 组合 | 功能强大，蓝图可视化 | 粒度太细，容易产生「一千个哈姆雷特」 |
| **MMORPG 方案** | 腾讯 MMORPG | 业务层抽象：入口、同步、流程控制、表现 | 学习门槛低，类型收敛 | 需要与策划共同定义抽象粒度 |

### A.2 抽象粒度的选择

这是设计技能系统时最关键的问题：

```
┌─────────────────────────────────────────────────────────────────────┐
│                      抽象粒度的权衡                                  │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│   粒度太细（UE GAS 风格）：                                         │
│   ┌─────────────────────────────────────────────────────────────┐  │
│   │  播放动画 Task + 播放特效 Task + 播放音效 Task + ...        │  │
│   │                                                             │  │
│   │  优点：灵活到极致                                           │  │
│   │  缺点：不同策划配置出完全不同的流程图，难以调试              │  │
│   └─────────────────────────────────────────────────────────────┘  │
│                                                                      │
│   粒度适中（MMORPG 业务抽象）：                                     │
│   ┌─────────────────────────────────────────────────────────────┐  │
│   │  动作节点（内置动画+特效+音效+震屏）                        │  │
│   │  投射物节点（内置轨迹+命中检测）                            │  │
│   │  位移节点（内置位移+碰撞）                                  │  │
│   │                                                             │  │
│   │  优点：策划易于理解，流程一致                               │  │
│   │  缺点：某些边界情况需要额外处理                             │  │
│   └─────────────────────────────────────────────────────────────┘  │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

### A.3 技能流程的节点分类

根据 MMORPG 技能管线设计的经验，技能流程节点可以分为：

| 节点类型 | 说明 | 示例 |
|----------|------|------|
| **入口节点** | 技能开始的条件 | 玩家主动释放、AI 触发 |
| **请求节点** | 客户端-服务器交互 | 选目标、结算效果、同步 |
| **流程控制节点** | 控制流程走向 | 条件分支、循环、随机 |
| **表现节点** | 视觉效果 | 动作、特效、音效、震屏 |

**设计原则**：
- **成长性数值**（伤害、冷却等）保留在 Excel/配置表中，策划可以随时调整
- **技能流程**（阶段、时间轴、效果组合）通过蓝图/Pipeline 配置，数据驱动
- **效果实现**（具体怎么造成伤害）由程序实现，策划只负责组合

### A.6 同步策略

```
┌─────────────────────────────────────────────────────────────────────┐
```

### 2.3 框架解决方案：Pipeline（能力管线）

**Pipeline** 就是解决「技能流程编排」问题的模块：

```csharp
// Pipeline 示例：火球术
public class FireballAbilityPipeline : AbilityPipeline<SkillContext>
{
    protected override void Build()
    {
        // 前摇阶段：播放抬手动画，持续 0.5 秒
        PreCast = new DurationalPhase(duration: 0.5f);

        // 施法阶段：时间轴阶段
        Cast = new TimelinePhase()
            .OnEnter(ctx => {
                // 进入施法：消耗魔法值
                ctx.Caster.Mp -= 50;
            })
            .AddEvent(atMs: 0, () => {
                // 0ms：发射火球
                SpawnProjectile(ctx);
            })
            .AddEvent(atMs: 100, () => {
                // 100ms：播放音效
                PlaySound(ctx, "fireball_cast");
            })
            .AddEvent(atMs: 300, () => {
                // 300ms：造成伤害
                DealDamage(ctx);
            })
            .OnExit(ctx => {
                // 退出施法：进入冷却
                ctx.Cooldown = 10f;
            });
    }
}
```

**Pipeline 的核心概念**：

| 概念 | 说明 |
|------|------|
| `AbilityPipeline` | 技能管线的定义（蓝图） |
| `AbilityPipelineRun` | 技能管线的执行实例（一次运行） |
| `Phase` | 管线中的单个阶段 |
| `TimelinePhase` | 时间轴阶段，支持在精确时间点触发事件 |

## 4. 被动技能：事件的「监听与响应」

### 3.1 被动技能的本质

主动技能是「玩家告诉技能做什么」，被动技能是「技能自己感知环境并响应」：

```
│   └─────────┘      └─────────┘      └─────────┘                   │
│                                                                      │
│   被动技能：                                                          │
│   ┌─────────┐      ┌─────────┐      ┌─────────┐                   │

│   └─────────┘      └─────────┘      └─────────┘                   │
│                                                                      │
│   例子：                                                              │
```

### 3.2 被动技能的共性分析

```

│                      被动技能共性分析                                │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│   ┌──────────────────────────────────────────────────────────────┐  │
│   │                   被动技能 = 事件 + 条件 + 行为               │  │
│   │                                                              │  │
│   │   ┌─────────┐    ┌─────────┐    ┌─────────┐                │  │
│   │   │  事件   │ →  │  条件   │ →  │  行为   │                │  │
│   │   │  触发  │    │  判断  │    │  执行  │                │  │
│   │   └─────────┘    └─────────┘    └─────────┘                │  │
│   │                                                              │  │
│   │   • 事件：什么情况触发？（受到伤害、击杀、周期性）            │  │
│   │   • 条件：满足什么条件才执行？（血量<30%、有BUFF、随机）     │  │

│   └──────────────────────────────────────────────────────────────┘  │
│                                                                      │
│   ┌──────────────────────────────────────────────────────────────┐  │

│   │   • NOT（非）：不是 A                                        │  │

│   └──────────────────────────────────────────────────────────────┘  │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

### 3.3 框架解决方案：Triggering（触发器系统）

**Triggering** 就是解决「事件监听与条件响应」问题的模块：

```csharp
// Triggering 示例：低血量护盾被动
public class LowHealthShieldPassive : TriggerPlan
{
    // 监听什么事件
    public string Event => "OnHealthChanged";

    // 什么条件下触发
    public ICondition Condition => new All
    {
        // 条件1：生命值低于30%
        new CompareCondition(
            left: new ContextField("CurrentHealthPercent"),
            op: CompareOp.LessThan,
            right: new Const(0.3f)
        ),
        // 条件2：当前没有护盾
        new Not
        {
            Condition = new HasTag("Shield")
        }
    };

    // 触发后执行什么
    public IAction[] Actions => new[]
    {
        // 行为1：添加护盾效果
        new ApplyEffect { EffectId = ShieldEffectId },
        // 行为2：设置冷却标记
        new SetContextValue { Key = "ShieldOnCooldown", Value = true },
        // 行为3：播放特效
        new TriggerPresentation { PresentationId = "shield_activate" }
    };
}
```

**Triggering 的核心概念**：

| 概念 | 说明 |
|------|------|
| `EventBus` | 事件总线，接收和分发游戏事件 |
| `TriggerRunner` | 触发器管理器，维护所有触发器 |
| `TriggerPlan` | 触发计划，定义「事件+条件+行为」 |
| `Predicate/Condition` | 条件判断 |
| `Action` | 具体行为执行 |

## 5. Buff/Debuff：时间的「魔法」

### 5.1 Buff 的本质

Buff 是「在一段时间内，持续产生效果」：

```
┌─────────────────────────────────────────────────────────────────────┐
│                         Buff 的生命周期                              │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│   ┌─────────────────────────────────────────────────────────────┐  │
│   │                        Buff 生命周期                         │  │
│   │                                                              │  │
│   │    Apply ──→ Active ──→ Tick ──→ Tick ──→ ... ──→ Remove  │  │
│   │      │                                                        │  │
│   │      │                                                        │  │
│   │      ▼                                                        │  │
│   │   ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐        │  │
│   │   │  应用   │  │  激活   │  │  周期   │  │  移除   │        │  │
│   │   │OnApply  │→ │OnActive │→ │ OnTick  │→ │OnRemove │        │  │
│   │   └─────────┘  └─────────┘  └────┬────┘  └─────────┘        │  │
│   │                                    │                          │  │
│   │                                    └────── 每隔一段时间 ─────┘  │
│   │                                                              │  │
│   └─────────────────────────────────────────────────────────────┘  │
│                                                                      │
│   例子：持续5秒的灼烧，每秒造成10点伤害                              │
│   • OnApply：标记目标被灼烧                                          │
│   • OnTick（每秒）：扣除10点生命                                     │
│   • OnRemove：移除灼烧标记                                           │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

### 5.2 Buff 的类型

```
┌─────────────────────────────────────────────────────────────────────┐
│                         Buff 类型分类                                 │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│   按持续时间：                                                        │
│   ├── 瞬时（Instant）：立即生效，立即移除                             │
│   ├── 持续（Duration）：持续一段时间                                  │
│   └── 永久（Infinite）：不自动移除，需要手动移除                      │
│                                                                      │
│   按触发方式：                                                        │
│   ├── 被动（Passive）：持续生效，不主动触发                          │
│   ├── 周期（Periodic）：每隔一段时间触发                              │
│   │   ├── DOT（持续伤害）                                           │
│   │   └── HOT（持续治疗）                                           │
│   └── 触发（Triggered）：特定事件触发                                │
│                                                                      │
│   按叠加方式：                                                        │
│   ├── 覆盖（Override）：新Buff替换旧Buff                            │
│   ├── 叠加（Stack）：增加层数                                        │
│   └── 独立（Separate）：多个Buff同时存在                             │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

### 5.3 框架解决方案：Effect 组件 + Schedule

**Effect** 解决 Buff 的「效果定义」问题，**Schedule** 解决「时间调度」问题：

```csharp
// Effect 示例：灼烧效果
public class BurningEffect : GameplayEffectSpec
{
    public BurningEffect()
    {
        DurationPolicy = EffectDurationPolicy.Duration;
        DurationSeconds = 5f;
        PeriodSeconds = 1f;  // 每秒触发一次

        Components = new IEffectComponent[]
        {
            // 效果组件1：属性修改（造成伤害）
            new AttributeEffectComponent(
                new AttributeEffect(
                    TargetAttribute: "Health",
                    Operation: ModifierOperation.Add,
                    Magnitude: -10f
                )
            ),
            // 效果组件2：触发事件（用于播放特效）
            new TriggerEventEffectComponent("OnBurnTick")
        };
    }
}
```

```csharp
// Schedule 示例：注册一个周期性调度任务
var scheduleManager = world.Services.GetService<IScheduleManager>();

// 注册灼烧效果：每秒执行一次，最多5次
var handle = scheduleManager.RegisterPeriodic(
    intervalMs: 1000,           // 间隔1秒
    maxExecutions: 5,          // 最多执行5次
    businessId: burningBuffId, // 关联Buff ID
    effect: new BurningEffectCallback()
);

// BurningEffectCallback 实现 IScheduleEffect
public class BurningEffectCallback : IScheduleEffect
{
    public void Execute(in ScheduleContext ctx)
    {
        // 执行伤害
        ctx.Target.ModifyAttribute("Health", -10);

        // 触发事件（播放灼烧特效）
        ctx.EventBus.Publish("OnBurnTick", ctx);
    }
}
```

### 6.3 复杂实体管理：召唤物、载具与跟随系统

除了简单的 Buff 和投射物，现代游戏还有大量「复杂实体」需求：

```
┌─────────────────────────────────────────────────────────────────────┐
│                    复杂实体需求的典型场景                             │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│   召唤物：                                                          │
│   • 召唤一个骷髅战士，骷髅有自己的 AI、攻击、技能                    │
│   • 骷髅跟随主人移动，主人死亡时骷髅也消失                          │

│   • 分身模仿本体的技能，但伤害降低                                   │

│                                                                      │

│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

### 6.3.1 复杂实体的本质分析

这些需求的共同点是什么？

```
┌─────────────────────────────────────────────────────────────────────┐
│                    复杂实体的本质                                   │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│   ┌─────────────────────────────────────────────────────────────┐  │
│   │                    复杂实体 = 生命周期管理                    │  │
│   │                                                             │  │

│                                                                      │
```

### 6.3.2 复杂实体 vs 普通实体

| 对比项 | 普通实体 | 复杂实体 |
|--------|----------|----------|
| **生命周期** | 独立 | 依赖其他实体 |
| **行为模式** | 固定 | 可配置 |
| **与系统交互** | 被动 | 主动 |
| **典型例子** | 子弹、特效 | 召唤物、载具、分身 |
| **管理复杂度** | 低 | 高 |

### 6.3.6 载具/坐骑系统

### 6.4 框架解决方案：Condition 组合 + Action 序列

**条件组合**：

```csharp
// 条件组合示例
public ICondition GetCondition()
{
    // (生命值 < 30%) AND (NOT 处于隐身状态) AND (目标有 "易伤" 标记)
    return new All
    {
        Items = new ICondition[]
        {
            // 条件1：生命值 < 30%
            new CompareCondition
            {
                Left = new BlackboardValue("Self.HealthPercent"),
                Operator = CompareOp.LessThan,
                Right = new Const(0.3f)
            },
            // 条件2：NOT 隐身
            new Not
            {
                Condition = new HasTag("Invisible")
            },
            // 条件3：目标有 "易伤" 标记
            new HasTag("Vulnerable")
        }
    };
}
```

**行为序列**：

```csharp
// Action 序列示例
public IAction[] GetActions()
{
    return new IAction[]
    {

        },
        // 行为2：眩晕目标（1秒）
        new ApplyEffect
        {
            EffectId = StunEffectId,
            Duration = 1f
        },
        // 行为3：造成伤害
        new DamageAction
        {
            Damage = 100,
            DamageType = "Physical"
        }
    };
}
```

## 8. 效果复用：组件的「组合」

### 8.1 属性修改器（Modifier）：可还原的数值变更

### 8.1.1 为什么需要修改器？

在技能系统中，很多属性修改是需要「可还原」的：

```
┌─────────────────────────────────────────────────────────────────────┐
│                    属性修改的「持久化」问题                           │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│   场景1：攻击力 +100（来自 BUFF）                                    │
│   ─────────────────────────────────                                 │
│   • 施放「力量祝福」技能                                            │
│   • 攻击力 +100，持续 10 秒                                         │
│   • 10 秒后，攻击力需要恢复原值                                    │
│                                                                      │
│   问题：这个 +100 怎么存储？怎么还原？                               │
│                                                                      │
│   ┌─────────────────────────────────────────────────────────────┐  │
│   │  错误做法：直接修改属性值                                    │  │
│   │  ┌─────────────────────────────────────────────────────────┐ │  │
│   │  │  health = 100;        // 原始值                          │ │  │
│   │  │  health += 50;        // BUFF +50                       │ │  │
│   │  │  health -= 30;        // 受伤 -30                       │ │  │
│   │  │  // BUFF 结束了，我怎么知道要 -50？                     │ │  │
│   │  │  health -= ???;       // 不知道该减多少了！             │ │  │
│   │  └─────────────────────────────────────────────────────────┘ │  │
│   └─────────────────────────────────────────────────────────────┘  │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

### 8.1.2 修改器的本质

修改器（Modifier）是一种「带来源标识的数值修改」：

```
┌─────────────────────────────────────────────────────────────────────┐
│                      修改器的核心思想                                │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│   修改器 = 数值变化 + 来源标识 + 操作类型                           │
│                                                                      │
│   ┌─────────────────────────────────────────────────────────────┐  │
│   │  struct Modifier                                            │  │
│   │  {                                                          │  │
│   │      ModifierSourceId Source;    // 谁加的？                 │  │
│   │      string Attribute;           // 修改哪个属性？           │  │
│   │      ModifierOperation Op;      // 加/减/乘/覆盖？         │  │
│   │      float Magnitude;           // 改变多少？               │  │
│   │  }                                                          │  │
│   └─────────────────────────────────────────────────────────────┘  │
│                                                                      │
│   示例：                                                            │
│   • Modifier { Source: Buff_001, Attribute: "Attack", Op: Add, Mag: +100 }  │
│   • Modifier { Source: Buff_002, Attribute: "Attack", Op: Add, Mag: +50 }   │
│   • Modifier { Source: Skill_003, Attribute: "Attack", Op: Mult, Mag: ×1.5 }│
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

### 8.1.3 修改器的计算流程

```
┌─────────────────────────────────────────────────────────────────────┐
│                    修改器的计算流程                                  │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│   BaseValue = 100  （角色基础攻击力）                               │
│                                                                      │
│   ┌─────────────────────────────────────────────────────────────┐  │
│   │                   修饰器计算器（ModifierCalculator）          │  │
│   │                                                             │  │
│   │   收集所有修改器：                                           │  │
│   │   ┌─────────────────────────────────────────────────────┐  │  │
│   │   │  Mod_Add: +100 (Buff_001)                           │  │  │
│   │   │  Mod_Add: +50  (Buff_002)                          │  │  │
│   │   │  Mod_Mult: ×1.5 (Skill_003)                        │  │  │
│   │   └─────────────────────────────────────────────────────┘  │  │
│   │                                                             │  │
│   │   计算过程：                                                │  │
│   │   Step 1: Base = 100                                       │  │
│   │   Step 2: Add 叠加 = 100 + 50 = 150                       │  │
│   │   Step 3: Mult 叠加 = 150 × 1.5 = 225                     │  │
│   │                                                             │  │
│   │   FinalValue = 225                                        │  │
│   └─────────────────────────────────────────────────────────────┘  │
│                              │                                        │
│                              ▼                                        │
│   ┌─────────────────────────────────────────────────────────────┐  │
│   │  Buff_002 结束 → 移除 +50 的修改器                          │  │
│   │  重新计算：100 + 100 = 200                                 │  │
│   └─────────────────────────────────────────────────────────────┘  │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

### 8.1.4 修改器操作类型

| 操作类型 | 说明 | 示例 |
|----------|------|------|
| **Add** | 加法叠加 | 攻击力 +100，多个叠加 |
| **Mult** | 乘法叠加 | 伤害 ×1.5，多个叠加 |
| **Override** | 直接覆盖 | 无视护甲设置为 0 |
| **PercentAdd** | 百分比加法 | 生命值 +10%（先算 Add，再算 PercentAdd） |

### 8.1.5 修改器 vs 直接修改

| 对比项 | 直接修改 | 修改器 |
|--------|----------|--------|
| **持久性** | 一次性 | 可还原 |
| **来源追踪** | 无法追踪 | 有 SourceId |
| **叠加规则** | 手动处理 | 内置多种叠加 |
| **适用场景** | 一次性伤害/治疗 | Buff/被动技能/光环 |

**原则**：所有需要「还原」的属性修改，都应该用修改器。

### 8.2 标签系统（Tags）：状态标识

### 8.2.1 为什么需要标签系统？

技能系统中需要判断各种「状态」：

```
┌─────────────────────────────────────────────────────────────────────┐
│                    标签系统的作用                                    │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│   场景1：沉默状态下不能施法                                         │
│   ─────────────────────────────────                                 │
│   if (HasTag("Silence")) return;  // 有沉默标签？不能施法           │
│                                                                      │
│   场景2：无敌状态下不受到伤害                                       │
│   ─────────────────────────────────                                 │
│   if (HasTag("Invincible")) return;  // 有无敌标签？不受伤害        │
│                                                                      │
│   场景3：变形状态下使用特殊技能                                      │
│   ─────────────────────────────────                                 │
│   if (HasTag("BearForm")) {       // 熊形态？                      │
│       skill = "熊击";             // 使用熊形态专属技能              │
│   }                               //                                   │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

### 8.2.2 标签 vs 属性

| 对比项 | 属性（Attribute） | 标签（Tag） |
|--------|-------------------|-------------|
| **值类型** | 数值 | 布尔（有/无） |
| **用途** | 数值计算 | 状态判断 |
| **示例** | Health=100, Attack=50 | "Silence", "Invincible", "BearForm" |
| **判断** | >, <, == | HasTag() |
| **修改** | +、-、×、覆盖 | AddTag(), RemoveTag() |

### 8.2.3 标签的类型分类

```
┌─────────────────────────────────────────────────────────────────────┐
│                       标签类型分类                                  │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│   控制类标签（硬控制）：                                             │
│   • Silence（沉默）── 不能施放技能                                  │
│   • Stun（眩晕）── 不能移动和施法                                  │
│   • Fear（恐惧）── 强制向某个方向移动                              │
│   • Freeze（冰冻）── 完全冻结                                      │
│                                                                      │
│   软控制类标签：                                                    │
│   • Slow（减速）── 移动速度降低                                    │
│   • Silence（沉默）── 不能施法                                     │
│   • Blind（致盲）── 命中率降低                                     │
│                                                                      │
│   状态类标签：                                                      │
│   • Invincible（无敌）── 不受伤害                                  │
│   • Invisible（隐身）── 不可被选中                                 │
│   • Shield（护盾）── 有护盾保护                                    │
│                                                                      │
│   形态类标签：                                                      │
│   • BearForm（熊形态）── 变身后使用不同技能                        │
│   • Stealth（潜行）── 隐身移动                                     │
│                                                                      │
│   来源类标签：                                                      │
│   • Buff_From_Boss（来自Boss的Buff）── 用于判断来源               │
│   • Damage_From_Fire（来自火焰的伤害）── 用于元素反应              │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

### 8.2.4 标签的叠加规则

```
┌─────────────────────────────────────────────────────────────────────┐
│                    标签的叠加规则                                    │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│   ┌─────────────────────────────────────────────────────────────┐  │
│   │                      独立标签（最常见）                      │  │
│   │                                                             │  │
│   │   同一个目标可以有多个独立标签，互不影响                      │  │
│   │   HasTag("Stun") = true                                    │  │
│   │   HasTag("Slow") = true                                    │  │
│   │   // 眩晕的同时被减速                                       │  │
│   └─────────────────────────────────────────────────────────────┘  │
│                                                                      │
│   ┌─────────────────────────────────────────────────────────────┐  │
│   │                      互斥标签                                │  │
│   │                                                             │  │
│   │   同一类型的标签只能存在一个，新标签替换旧标签              │  │
│   │   // 变形状态：熊形态 和 人形态 互斥                        │  │
│   │   AddTag("BearForm") → RemoveTag("HumanForm")              │  │
│   └─────────────────────────────────────────────────────────────┘  │
│                                                                      │
│   ┌─────────────────────────────────────────────────────────────┐  │
│   │                      可叠加标签                             │  │
│   │                                                             │  │
│   │   同一标签可以有多层，用于表示强度                          │  │
│   │   AddTag("Poison", stack=3)                                │  │
│   │   // 中毒3层，每层每秒-5伤害                                │  │
│   │   // 叠满5层时触发「剧毒」效果                              │  │
│   └─────────────────────────────────────────────────────────────┘  │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

### 8.2.6 框架中的标签系统

```csharp
// 标签检查
public bool HasTag(string tag) => _tags.Contains(tag);
public bool HasAnyTag(params string[] tags) => tags.Any(t => _tags.Contains(t));
public bool HasAllTags(params string[] tags) => tags.All(t => _tags.Contains(t));

// 标签操作
public void AddTag(string tag, int stack = 1);
public void RemoveTag(string tag);
public void ClearTags();

// 配合修改器的示例
public void OnBuffApplied(Buff buff)
{
    // 添加标签
    foreach (var tag in buff.Tags)
    {
        AddTag(tag);
    }

    // 应用属性修改器
    foreach (var modifier in buff.Modifiers)
    {
        ApplyModifier(modifier);
    }
}

public void OnBuffRemoved(Buff buff)
{
    // 移除标签
    foreach (var tag in buff.Tags)
    {
        RemoveTag(tag);
    }

    // 移除属性修改器（通过 SourceId 精确移除）
    foreach (var modifier in buff.Modifiers)
    {
        RemoveModifier(modifier.SourceId);
    }
}
```

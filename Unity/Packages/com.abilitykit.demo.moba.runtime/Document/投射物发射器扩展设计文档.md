# MOBA 投射物发射器扩展设计文档

> 阅读对象：需要扩展 MOBA 投射物发射流程、设计复杂弹幕/蓄力/准备态发射/多阶段发射的开发者。
>
> 文档目标：说明当前投射物发射器为什么拆成持续行为、发射上下文、发射序列和发射形状提供器，以及后续面对复杂发射需求时应该在哪一层扩展。

---

## 一、背景与问题

早期投射物发射流程主要集中在 `MobaProjectileService` 中，流程大致是：

1. 读取发射器配置。
2. 创建 launcher actor。
3. 构造 `ProjectileSpawnParams`。
4. 根据每次发射数量和扇形角创建 pattern。
5. 调用 projectile service 创建 schedule。
6. 在中断或清理时取消 schedule。

这种方式可以支持简单的单发、扇形和间隔发射，但随着需求增加会出现明显问题：

- 发射器配置不同会导致 service 中条件分支越来越多。
- 发射时序和发射形状混在一起，难以组合。
- “先实例化在身后准备，再往前释放”这类需求不是普通 spawn 能表达的。
- 发射过程本身需要可中断，但低层 projectile schedule 只负责发射调度，不负责业务生命周期。
- 修改器、launcher 归属、source context、skill runtime retain 等业务关联容易散落在不同分支中。

因此，发射器不应该只是一次 service 调用，而应该是一个可被持续行为系统管理的业务运行时对象。

---

## 二、设计目标

当前发射器扩展设计遵循以下目标：

- 发射过程是一个可中断持续行为。
- 发射时序、发射形状、准备态、释放策略互相解耦。
- 默认逻辑保持兼容，不破坏当前 `DurationMs`、`IntervalMs`、`CountPerShot`、`FanAngleDeg` 的简单配置。
- 新增复杂需求时优先新增局部策略，不继续撑大 `MobaProjectileService`。
- 发射过程中的 source context、modifier 查询、launcher actor 归属、schedule 清理仍有统一入口。

---

## 三、总体架构

当前发射器流程拆成五层：

```text
技能/效果触发
    |
    v
MobaProjectileService
    |  构建发射请求、launcher actor、base spawn、发射上下文
    v
MobaProjectileLaunchContinuous
    |  接入 IContinuous 生命周期：Activate / Tick / Abort / End
    v
IMobaProjectileLaunchSequence
    |  管发射时序：立即、间隔、延迟、准备后释放、多阶段时间线
    v
IProjectileSpawnPatternProvider
    |  管每次发射形状：单发、爆发、扇形、环形、矩阵、多目标
    v
IProjectileService
       负责底层 projectile spawn / schedule / cancel
```

核心原则是：

| 层级 | 负责内容 | 不负责内容 |
| --- | --- | --- |
| `MobaProjectileService` | 校验、创建 launcher actor、构建上下文 | 不写复杂发射分支 |
| `MobaProjectileLaunchContinuous` | 持续行为生命周期、中断入口 | 不决定具体发射时序 |
| `IMobaProjectileLaunchSequence` | 发射过程和时序 | 不关心技能触发入口 |
| `IProjectileSpawnPatternProvider` | 每次发射的形状和数量 | 不管理持续生命周期 |
| `IProjectileService` | 底层 projectile 生成、调度、取消 | 不承载 MOBA 业务策略 |

## 四、关键类型说明

### 4.1 `MobaProjectileLaunchRequest`

`MobaProjectileLaunchRequest` 是发射请求数据边界，记录一次发射所需的基础输入：

- 施法者 actor id。
- 发射器配置 `ProjectileLauncherMO`。
- 子弹配置 `ProjectileMO`。
- 本次已解析的 `CountPerShot` 和 `FanAngleDeg`。
- 发射位置和方向。
- `ProjectileSourceContext`。

它不持有 runtime service，也不直接执行任何逻辑。

### 4.2 `MobaProjectileLaunchContinuous`

`MobaProjectileLaunchContinuous` 是真正接入持续行为系统的运行时对象。

职责：

- `Activate` 时调用发射执行器启动发射。
- `TickManaged` 时检查发射是否完成。
- `Abort` 或 `End` 时通知 sequence 停止发射。
- 向运行时查询服务提供 debug 信息。

这层让发射器发射过程可以被 `IContinuousManager` 管理，从而支持后续统一中断、暂停、恢复、owner 查询等能力。

### 4.3 `MobaProjectileLaunchContext`

`MobaProjectileLaunchContext` 是 sequence 执行所需的上下文，包含：

- 原始 request。
- launcher / projectile 配置。
- launcher actor id 和 launcher entity。
- `ProjectileSpawnParams`。
- launcher source context。
- start frame、end time、interval frames、repeat count、bullets per shot。
- `IProjectileService`、`MobaProjectileLinkService`、`MobaSkillParamModifierService`。
- `IMobaProjectileLaunchRuntime`。

它的作用是把 service 中分散的数据收束成稳定边界。后续新增 sequence 时，优先消费 context，而不是回头依赖整个 `MobaProjectileService`。

### 4.4 `IMobaProjectileLaunchSequence`

`IMobaProjectileLaunchSequence` 是发射过程扩展点。

接口职责：

```csharp
public interface IMobaProjectileLaunchSequence
{
    bool TryStart(in MobaProjectileLaunchContext context, out MobaProjectileLaunchResult result);
    bool IsComplete(in MobaProjectileLaunchResult result);
    void Stop(in MobaProjectileLaunchResult result, ContinuousEndReason reason);
}
```

不同发射器的时序差异应该落在这里，例如：

- `RepeatProjectileLaunchSequence`：持续间隔发射。
- `DelayedProjectileLaunchSequence`：延迟一段时间后发射。
- `PrepareThenReleaseProjectileLaunchSequence`：先创建准备态对象，再释放。
- `TimelineProjectileLaunchSequence`：按时间线多个事件发射。
- `ChannelProjectileLaunchSequence`：引导期间持续发射，中断立即停止。

### 4.5 `MobaModifierProjectileSpawnPatternProvider`

`MobaModifierProjectileSpawnPatternProvider` 是当前默认 pattern provider，负责在每次 projectile schedule 触发时重新解析 modifier 驱动的发射数量和扇形角。

它保留了现有逻辑：

- 单发：`SingleShotPattern`。
- 多发无角度：`BurstPattern`。
- 多发有角度：`FanPattern`。

后续复杂形状不应该塞进 service，而应该新增独立 pattern provider 或 pattern factory。

## 五、当前默认流程

当前 `RepeatProjectileLaunchSequence` 覆盖原有发射行为，流程为：

1. `MobaProjectileService.LaunchFromSpawn` 创建 `MobaProjectileLaunchRequest`。
2. 创建 `MobaProjectileLaunchContinuous` 并交给 `IContinuousManager.TryActivate`。
3. continuous 激活后调用 `MobaProjectileService.TryStartLaunch`。
4. service 校验依赖、创建 launcher actor、构造 `ProjectileSpawnParams`。
5. service 创建 `MobaProjectileLaunchContext`。
6. service 选择默认 `RepeatProjectileLaunchSequence`。
7. sequence 绑定 launcher source。
8. sequence 创建 `MobaModifierProjectileSpawnPatternProvider`。
9. sequence 调用 `IProjectileService.ScheduleEmit`。
10. sequence 给 launcher entity 添加 `ProjectileLauncherComponent`。
11. continuous 每帧 tick，委托 sequence 检查是否完成。
12. 中断时 continuous 调用 sequence stop，sequence 取消 schedule 并推进 launcher 结束时间。
13. launcher cleanup 系统在 active bullets 归零后销毁 launcher actor。

## 六、如何扩展不同需求

### 6.1 间隔发射

继续使用 `RepeatProjectileLaunchSequence`。

适用场景：

- 每 N 毫秒发射一次。
- 持续一段时间的弹幕。
- 每次发射一枚或多枚。
- 数量和角度需要被 modifier 动态影响。

配置仍可使用当前字段：

- `DurationMs`
- `IntervalMs`
- `CountPerShot`
- `FanAngleDeg`

### 6.2 扇形、爆发、环形、矩阵发射

形状差异应该优先扩展 pattern provider。

推荐新增：

```csharp
public sealed class RingProjectileSpawnPatternProvider : IProjectileSpawnPatternProvider
{
    public IProjectileSpawnPattern GetPattern(in ProjectileSpawnParams baseSpawn, int frame)
    {
        // 根据配置返回 RingPattern 或自定义 pattern
    }
}
```

如果只是“同一时刻生成多枚 projectile，方向/位置不同”，通常属于 pattern。

### 6.3 先在后方准备，再向前发射

这种需求不应当只是一个 pattern。它包含明显阶段：

1. Prepare：创建准备态 actor 或 visual。
2. Hold：准备态对象停留、跟随、旋转、蓄力或等待释放。
3. Release：正式生成 projectile 或把准备态对象切换为 projectile。
4. Cleanup：中断或释放后清理准备态对象。

推荐新增 `PrepareThenReleaseProjectileLaunchSequence`。

伪流程：

```text
TryStart:
  创建 launcher actor
  创建若干准备态 actor/visual
  记录准备态列表到 result 或 sequence runtime state

Tick:
  未到 release frame：更新准备态位置
  到 release frame：逐个释放 projectile
  所有 projectile 释放且 active bullets 归零：完成

Stop:
  取消未释放 projectile
  销毁或回收准备态 actor/visual
  推进 launcher 结束时间
```

如果准备态对象需要网络同步，建议作为 actor entity 走实体生成模块；如果只是纯表现，可以只在 view 层处理，但逻辑层仍应保留可中断的 release 时序。

### 6.4 多阶段弹幕

多阶段弹幕建议新增 `TimelineProjectileLaunchSequence`。

配置可以表达为：

```text
Timeline
  Event 0ms: FanPattern, projectile 1001, count 3
  Event 300ms: RingPattern, projectile 1002, count 12
  Event 700ms: LinePattern, projectile 1003, count 5
```

sequence 在 tick 时按当前帧/时间触发事件。每个事件可以选择不同 pattern、projectile、spawn offset 或 target policy。

### 6.5 引导发射与可打断发射

引导类技能应让发射 sequence 持有“是否仍可继续”的判断，continuous 只负责统一中断入口。

推荐规则：

- 技能被硬控打断：调用 `IContinuousManager.TryInterrupt`。
- 玩家主动取消：调用 `TryInterrupt` 或技能系统统一 cancel。
- 引导自然结束：sequence `IsComplete` 返回 true。
- 中断时未发出的 schedule 或准备态对象必须在 sequence `Stop` 中处理。

## 七、配置演进建议

当前 `ProjectileLauncherDTO` 字段较少，适合默认 repeat 行为：

```text
ProjectileLauncherDTO
  Id
  Name
  EmitterType
  DurationMs
  IntervalMs
  CountPerShot
  FanAngleDeg
```

后续建议不要无限追加具体业务字段，而是改为引用子配置：

```text
ProjectileLauncherDTO
  Id
  Name
  EmitterType
  SequenceId
  PatternId
  PrepareId
  PolicyId
  DefaultProjectileId
```

建议配置拆分：

| 配置 | 职责 |
| --- | --- |
| `LaunchSequenceConfig` | 发射时序：repeat、delay、timeline、prepare-then-release |
| `LaunchPatternConfig` | 发射形状：single、fan、ring、line、grid、target-list |
| `ProjectilePrepareConfig` | 准备态布局、跟随规则、释放规则 |
| `ProjectileReleasePolicyConfig` | 释放方向、锁定、追踪、目标重选 |
| `ProjectileMO` | projectile 飞行、碰撞、命中、生命周期 |

短期可以保留旧字段作为默认兼容：

- 如果 `SequenceId` 为空，使用 `RepeatProjectileLaunchSequence`。
- 如果 `PatternId` 为空，使用当前 count/fan 默认 pattern。
- 如果 `PrepareId` 为空，不进入准备态。

## 八、是否接入 Pipeline 模块

复杂发射需求可以接入 `com.abilitykit.pipeline`，但不建议把所有发射逻辑都直接改成管线驱动。

当前发射器已经有明确生命周期分层：

- `MobaProjectileLaunchContinuous` 负责接入持续行为、中断和结束。
- `IMobaProjectileLaunchSequence` 负责发射过程和时序。
- `IProjectileSpawnPatternProvider` 负责单次发射形状。
- `IProjectileService` 负责底层 projectile spawn / schedule / cancel。

如果让 pipeline 直接替代这套发射生命周期，容易出现两套生命周期同时管理同一件事：一套来自 `IContinuous`，一套来自 pipeline run。这样中断、完成、清理、debug 归属会变得不清晰。

推荐原则是：pipeline 作为可选的高阶编排层，而不是默认发射底座。

| 场景 | 是否接 Pipeline | 推荐落点 |
| --- | --- | --- |
| 单发、间隔发射、扇形、爆发 | 不需要 | `RepeatProjectileLaunchSequence` + pattern provider |
| 环形、矩阵、多方向、多目标 | 通常不需要 | 新增 `IProjectileSpawnPatternProvider` |
| 先准备、延迟释放、蓄力后释放 | 不一定需要 | 新增 `PrepareThenReleaseProjectileLaunchSequence` |
| 多阶段发射：准备、锁定、预警、分批释放、收束、二段爆发 | 可以接 | 新增 pipeline-backed launch sequence |
| 需要策划可视化编排、阶段调试、复用阶段节点 | 建议接 | 用 pipeline phase 表达内部阶段 |
| 技能整体流程：前摇、吟唱、消耗、发射、后摇、冷却 | 建议接 | 技能施法 pipeline 的某个 phase 触发 projectile launch |

推荐两种接入方式。

### 8.1 技能管线调用发射器

技能整体流程可以使用 pipeline 编排，例如：

```text
SkillCastPipeline
  Phase 1: 前摇/吟唱
  Phase 2: 条件检测与资源消耗
  Phase 3: 播放表现和音效
  Phase 4: 调用 MobaProjectileService.LaunchFromSpawn
  Phase 5: 后摇/冷却
```

这种方式下，pipeline 管的是技能施法流程；真正的发射器运行时仍由 `MobaProjectileLaunchContinuous` 和 sequence 管理。

优点是技能阶段清晰，且不会破坏 projectile 发射器的中断和清理边界。

### 8.2 发射器内部使用 pipeline-backed sequence

当一个发射器内部阶段非常复杂时，可以新增类似 `PipelineProjectileLaunchSequence` 的实现：

```text
PipelineProjectileLaunchSequence : IMobaProjectileLaunchSequence
  TryStart:
    创建发射专用 pipeline run
    把 MobaProjectileLaunchContext 包装为 pipeline context

  IsComplete:
    查询 pipeline run 是否完成

  Stop:
    interrupt pipeline run
    清理未释放 projectile、准备态 actor、临时绑定
```

这种方式下，外部仍然只看到一个 `IMobaProjectileLaunchSequence`。`MobaProjectileLaunchContinuous` 继续作为唯一中断入口；pipeline 只负责这个 sequence 内部的阶段编排。

适合的复杂示例可以是：

```text
PipelineProjectileLaunchSequence
  Phase 1: SpawnPrepareActors
    在角色身后生成 5 个准备态 projectile actor

  Phase 2: ChargeAndFollow
    持续 800ms，准备态对象跟随角色并逐渐展开

  Phase 3: LockTargets
    锁定多个目标或计算释放方向

  Phase 4: WarningPresentation
    生成预警区域，等待 300ms

  Phase 5: ReleaseWaveA
    第一波按扇形释放 3 个 projectile

  Phase 6: Delay
    等待 200ms

  Phase 7: ReleaseWaveB
    第二波按环形释放剩余 projectile

  Phase 8: Cleanup
    清理未释放准备态对象和临时绑定
```

后续真要补全复杂示例时，建议优先实现这个 pipeline-backed sequence，而不是改动 `MobaProjectileService` 或底层 `IProjectileService`。

### 8.3 不建议接入 pipeline 的情况

以下需求保持轻量扩展更合适：

- 只是数量、角度、形状变化。
- 只是简单 repeat / delay / burst。
- 只是 modifier 改变 count、fan angle、projectile id。
- 没有阶段复用、调试追踪或可视化编排需求。

这些需求继续落在 sequence / pattern provider 上即可，避免把简单发射变成过重的阶段图。

## 九、扩展落点判断表

| 需求 | 推荐扩展点 | 说明 |
| --- | --- | --- |
| 每隔 N 毫秒发射 | `IMobaProjectileLaunchSequence` | 发射时序变化 |
| 延迟后发射 | `IMobaProjectileLaunchSequence` | 发射时序变化 |
| 先准备再释放 | `IMobaProjectileLaunchSequence` + prepare state | 有阶段、有中断清理 |
| 扇形/环形/矩阵 | `IProjectileSpawnPatternProvider` | 单次发射形状变化 |
| 每次发射不同 projectile | `TimelineProjectileLaunchSequence` 或 sequence event | 时序事件选择 projectile |
| 根据目标列表发射 | pattern provider 或 target policy | 如果只是多方向生成，放 pattern；如果要选目标，放 policy |
| 中断后保留已发出的子弹 | sequence stop policy | 停止未发出部分，不处理已释放 projectile |
| 中断后全部消失 | sequence stop + projectile cleanup policy | 需要额外记录已释放 projectile id |
| 蓄力越久子弹越多 | sequence + modifier/charge context | sequence 根据 elapsed/charge 计算 pattern 参数 |
| 多阶段复杂编排 | pipeline-backed sequence | pipeline 只管 sequence 内部阶段，continuous 仍管外层生命周期 |

## 十、实现注意事项

### 10.1 不要把复杂逻辑继续写回 service

`MobaProjectileService` 应保持为入口和上下文构建器。新增需求如果需要修改大量 service 分支，通常说明扩展点选错了。

### 10.2 sequence 要负责自己的中断清理

凡是 sequence 创建的 schedule、准备态 actor、临时绑定，都应在 `Stop` 中有对应清理逻辑。

### 10.3 pattern 不应管理生命周期

pattern 只负责“这次发射生成哪些 projectile 参数”。它不应该关心 continuous 是否结束，也不应该销毁 launcher。

### 10.4 准备态对象要区分逻辑和表现

准备态如果影响命中、阻挡、网络同步，应作为逻辑 actor 管理。

准备态如果只是视觉效果，可以放到 view 层，但逻辑层仍要有 release 时间点和中断语义。

### 10.5 modifier 查询要保留 launcher 归属

当前 pattern provider 使用 root actor id 与 launcher actor id 构造 modifier resolve context。后续复杂 pattern 也应保留这条链路，避免丢失“角色 modifier”和“launcher scoped modifier”的区分。

## 十二、推荐后续迭代

第一阶段：配置引用化。

- 给 launcher 配置增加 `SequenceId`、`PatternId`、`PrepareId`。
- 保留旧字段作为默认兼容。
- 增加 sequence factory 和 pattern factory。

第二阶段：准备后释放。

- 新增 `PrepareThenReleaseProjectileLaunchSequence`。
- 定义准备态布局和释放策略。
- 中断时清理准备态对象。

第三阶段：复杂形状。

- 新增 ring、line、grid、target-list pattern provider。
- 将 pattern 配置表独立出来。

第四阶段：多阶段时间线。

- 新增 `TimelineProjectileLaunchSequence`。
- 支持一个发射器在不同时间点使用不同 projectile、pattern、offset、target policy。

第五阶段：pipeline-backed sequence 示例。

- 新增 `PipelineProjectileLaunchSequence`。
- 定义发射专用 pipeline context。
- 用阶段表达准备、蓄力、锁定、预警、多波释放和清理。
- 保持 `MobaProjectileLaunchContinuous` 作为唯一外层中断入口。

## 十三、总结

投射物发射器的本质不是一次 projectile spawn，而是一个有生命周期、可中断、可组合的发射过程。

当前设计将它拆成：

- continuous 管生命周期。
- service 管入口和上下文构建。
- sequence 管时序和阶段。
- pattern provider 管形状。
- projectile service 管底层实体生成和调度。

这样后续无论是间隔发射、复杂弹幕、准备态释放、蓄力、多阶段发射，都会落到清晰的局部扩展点上，而不是不断扩大 `MobaProjectileService` 的条件分支。
